# 首页对话记账 OCR 账单识别最佳实践方案

## 1. 文档目的

本文档基于当前项目现状，设计首页对话记账接入 OCR 账单识别的最佳实践方案，目标是让用户在首页通过：

- 上传图片
- 拍摄图片
- 图片 + 文字混合输入

即可完成账单识别，并继续沿用现有的“对话消息 + 确认卡片 + 入库”主链路。

本文档只给方案，不直接落代码。

---

## 2. 当前项目现状梳理

结合根目录文档、`docs` 文档和现有代码，当前状态如下：

- 首页核心交互已确定为“对话式记账”，文本输入后由服务端解析，返回确认卡片，再由用户确认入库。
- Server 端已有 `chat` 模块、LangGraph 记账编排图、本地降级解析、多笔账单确认能力。
- Mobile 端首页当前只支持文字输入和快捷输入，尚未接入拍照、相册选择、图片消息、OCR 识别入口。
- Server 端当前上传能力主要用于头像上传，`/upload/image` 默认写入公开 MinIO 地址，路径是 `avatars/...`，不适合直接复用于账单图片。
- 现有 `ChatMessage` 只有 `content`、`metadata`、`billId`，尚未抽象消息附件、OCR 任务、识别证据。
- 当前 `SendMessageDto` 只支持 `content`，不支持图片附件和混合输入。
- AI 记账文档里已明确图片记账属于目标能力，但当前代码仍停留在“文本解析优先”的阶段。

结论：

- 现有对话记账主链路是可复用的。
- 不能直接把账单 OCR 当成“头像上传复用”来做。
- 最优方案应新增“私有媒体 + OCR 编排 + 图片消息附件 + 结构化确认卡片增强”，而不是把图片识别逻辑硬塞进现有文本接口。

---

## 3. 设计目标

### 3.1 业务目标

- 用户在首页一键拍照或选图后，可以快速识别账单。
- OCR 识别结果最终仍回到对话流中的确认卡片，不破坏首页统一入口。
- 支持常见账单图片场景，而不是只支持一种“小票”。

### 3.2 技术目标

- 兼容当前 `mobile + server + MinIO + ChatService + LangGraph` 架构。
- 在不重构现有文本记账能力的前提下平滑扩展。
- 兼顾准确率、成本、隐私和后续扩展性。

### 3.3 体验目标

- 上传后 1 到 3 秒内给用户明确反馈，不能长时间无响应。
- 高置信度场景尽量减少编辑成本。
- 低置信度场景给出“可修正”的结构化结果，而不是一句“识别失败”。

---

## 4. 推荐方案总览

## 4.1 最优技术路线

推荐采用：

`图片预处理 -> OCR 专用服务 -> 文本归一化 -> 账单场景分类 -> 规则提取 -> LLM 结构化补全 -> 置信度评估 -> 确认卡片`

而不是直接采用：

`图片 -> 纯多模态 LLM -> 直接入库`

### 原因

- 中文支付截图、电子小票、纸质小票在“文本识别”阶段，专业 OCR 服务通常比纯多模态 LLM 更稳定。
- 账单识别本质上不是纯视觉问答，而是“文本抽取 + 结构化字段归一 + 账单规则判断”。
- 项目当前 LLM 编排已经偏文本解析，继续走“OCR 先抽文，再交给现有语义链路”改动最小。
- 后续如果要加入多模态 LLM，也应作为低质量 OCR 的兜底，而不是一期主路径。

### 最佳实践结论

- 一期主路径：`OCR 专用服务 + 现有 LangGraph/ChatService 扩展`
- 二期增强：`多模态 LLM 作为 OCR 低质量图像兜底`

## 5. 支持的账单图片范围

一期建议优先支持以下 4 类：

### 5.1 支付成功截图

如：

- 微信支付成功页
- 支付宝支付成功页
- 银行 App 支付成功页

特点：

- 识别成功率高
- 金额、时间、商户、支付方式通常比较明确
- 最适合作为 OCR 一期主场景

### 5.2 订单详情截图

如：

- 外卖订单页
- 电商订单详情页
- 打车订单详情页

特点：

- 文案结构较规则
- 金额字段可能有“原价/优惠/实付”多个候选
- 需要规则优先取“实付/支付金额”

### 5.3 纸质小票 / 收据

如：

- 餐饮小票
- 超市收银票
- 医院票据

特点：

- OCR 质量受拍摄角度、清晰度、光照影响大
- 推荐默认记录“一张票 = 一笔账单”，商品明细仅作为证据或备注，不拆成多笔流水

### 5.4 转账 / 入账截图

如：

- 工资到账
- 红包收入
- 转账收入
- 退款到账

特点：

- 需额外支持 `income`
- 不能默认全部当作 `expense`

不建议一期优先支持：

- 长截图多订单混排
- 复杂表格票据
- 模糊严重、遮挡严重图片
- 视频记账

## 6. 推荐交互方案

## 6.1 首页输入区改造

在现有首页底部输入区中增加两个入口：

- `图片`：从相册选择
- `拍照`：调用相机直接拍摄

推荐交互：

```text
[ 图片 ] [ 拍照 ] [ 输入框............... ] [ 发送 ]
```

说明：

- 仍保留文字输入主路径。
- 图片和文字可以同时存在，例如“这张是昨天晚饭，帮我记一下”。
- 发送后先展示用户图片消息气泡，再展示 AI 识别中的状态消息。

## 6.2 对话中的图片消息表现

用户发送图片后，对话中应展示：

- 图片缩略图
- 上传状态
- 识别状态

推荐状态机：

- `uploading`：上传中
- `processing`：OCR 识别中
- `confirm_card`：已生成确认卡片
- `failed`：识别失败，可重试

## 6.3 确认卡片设计建议

OCR 场景下确认卡片建议比文本卡片多展示两类信息：

- 识别来源：`图片识别`
- 识别证据：`商户 / 金额 / 日期 / 账户` 的来源提示

示例：

```text
美团外卖
支出 ¥32.00
餐饮 · 微信支付 · 昨天
来源：图片识别
证据：商户=美团外卖，金额=实付金额
[确认] [修改]
```

### 低置信度场景

如果金额识别不稳定，卡片中要明确提示：

- 识别到了商户，但金额不确定
- 识别到了总金额和实付金额两个候选，请用户选择
- 识别到了图片，但未判断是收入还是支出，请补充

这类卡片必须可编辑，不允许 silent fail。

## 7. 核心架构设计

## 7.1 总体链路

```text
Mobile 选择图片/拍照
    ->
客户端压缩与纠正方向
    ->
上传私有账单图片
    ->
创建用户消息（content + attachments）
    ->
Server OCR 编排
    ->
OCR 文字提取
    ->
账单场景分类
    ->
结构化字段提取
    ->
分类/账户匹配
    ->
置信度评估
    ->
生成 OCR 确认卡片
    ->
用户确认
    ->
创建 Bill
```

## 7.2 为什么要拆成两段能力

建议明确拆分为两个层：

### A. 媒体层

负责：

- 图片上传
- 图片压缩
- 私有存储
- 元数据记录
- 访问控制

### B. 识别层

负责：

- OCR
- 文本清洗
- 结构化字段提取
- 与账单领域模型对齐

这样做的好处：

- 上传和识别解耦
- 后续可以复用图片能力到头像、附件、账单证据、工单附件等场景
- OCR 厂商和 LLM 厂商可以独立替换

## 8. 最优识别策略

## 8.1 采用“规则优先 + LLM 补全”的混合方案

推荐执行顺序：

### Step 1. 图片类型分类

先判断图片属于哪一类：

- 支付成功页
- 订单详情页
- 纸质小票
- 收入截图
- 非账单图片

做法：

- 先根据 OCR 文本中的关键词做规则分类
- 规则不确定时再让 LLM 做场景判断

常见关键词：

- 支出：`支付成功`、`实付`、`付款方式`
- 收入：`已到账`、`收入`、`退款成功`
- 打车：`行程单`、`滴滴出行`
- 外卖：`美团外卖`、`饿了么`

### Step 2. OCR 文本抽取

优先使用专业 OCR 服务抽取：

- 全量文本
- 行块结构
- 坐标信息
- 关键字段候选

建议 OCR 输出不要只保留纯文本字符串，至少保留：

- `text`
- `lines`
- `boundingBoxes`
- `provider`
- `confidence`

因为后续做证据回显、质量分析、低置信度人工修正都需要这些信息。

### Step 3. 规则抽取关键字段

针对账单字段优先做规则提取：

- 金额
- 日期
- 商户
- 支付账户
- 收支类型

例如金额优先级：

1. `实付金额`
2. `支付金额`
3. `合计`
4. `总计`
5. `金额`

而不是简单取图片里最大的数字。

### Step 4. LLM 结构化补全

将 OCR 文本和规则提取结果一起交给 LLM，完成：

- 字段补全
- 分类推断
- 备注归一化
- 歧义解释
- 置信度辅助判断

LLM 输入建议包含：

- 原始 OCR 文本
- OCR 关键候选字段
- 图片场景分类结果
- 用户可用分类列表
- 用户可用账户列表
- 用户最近高频商户分类映射

### Step 5. 置信度评估

置信度不要只看 LLM 输出，要做多因子打分：

- OCR 整体置信度
- 金额字段是否命中强规则
- 场景分类是否明确
- 分类是否命中历史映射
- 日期是否明确
- 收支类型是否明确
- 是否存在多个冲突金额候选

推荐最终分级：

- `high`：金额、类型、分类、日期明确，冲突极低
- `medium`：金额明确，但分类或账户不够确定
- `low`：金额、类型或日期存在明显歧义

## 9. 为什么不建议“一张小票拆多笔账单”

对首页对话记账来说，最佳实践是：

- 默认一张票生成一笔账单
- 商品明细保留在 OCR 元数据或确认卡片详情中
- 不在一期将商品明细直接拆成多笔 `Bill`

原因：

- 用户真正记账关心的是一笔消费总额，而不是每个 SKU 一条流水。
- 多笔拆分会让确认卡片极度复杂，降低首页对话效率。
- 当前项目的 `Bill` 模型也更适合表示“记账事件”，不适合做“票据商品表”。

推荐策略：

- 账单主记录：`超市购物 ¥168`
- 元数据附加：`itemSummary = ["苹果", "牛奶", "纸巾"]`
- 后续如果做消费分析，再考虑商品级扩展

## 10. 数据模型建议

## 10.1 不建议直接复用当前头像上传模型

当前上传能力有两个问题：

- 路径固定为 `avatars/...`
- 公开访问 URL 不适合账单图片

账单图片属于敏感数据，最佳实践必须私有化。

## 10.2 推荐新增通用媒体表

建议新增 `MediaAsset`：

| 字段               | 说明                                            |
| ------------------ | ----------------------------------------------- |
| `id`               | 媒体 ID                                         |
| `userId`           | 上传用户                                        |
| `bucket`           | 存储桶                                          |
| `objectKey`        | 对象路径                                        |
| `mimeType`         | 文件类型                                        |
| `size`             | 文件大小                                        |
| `width` / `height` | 图片尺寸                                        |
| `sha256`           | 文件去重哈希                                    |
| `purpose`          | `avatar` / `chat_bill`                          |
| `visibility`       | `public` / `private`                            |
| `status`           | `uploaded` / `processing` / `ready` / `deleted` |
| `createdAt`        | 创建时间                                        |

## 10.3 推荐新增 OCR 任务表

建议新增 `OcrTask`：

| 字段               | 说明                                           |
| ------------------ | ---------------------------------------------- |
| `id`               | OCR 任务 ID                                    |
| `userId`           | 用户 ID                                        |
| `messageId`        | 关联对话消息                                   |
| `mediaAssetId`     | 关联图片                                       |
| `provider`         | OCR 服务商                                     |
| `sceneType`        | 截图/小票/收入单据等                           |
| `status`           | `pending` / `running` / `succeeded` / `failed` |
| `rawText`          | OCR 原始文本                                   |
| `rawResult`        | OCR 原始 JSON                                  |
| `structuredResult` | 结构化提取结果                                 |
| `confidenceScore`  | 最终置信度分数                                 |
| `errorMessage`     | 失败原因                                       |
| `createdAt`        | 创建时间                                       |
| `updatedAt`        | 更新时间                                       |

## 10.4 推荐增强 ChatMessage

当前 `ChatMessage` 建议最少补一个附件字段，方式有两种：

### 方案 A. 直接在 `metadata` 中挂附件

优点：

- 改动小

缺点：

- 后续扩展差
- 附件查询麻烦

### 方案 B. 新增 `ChatAttachment`

更推荐新增 `ChatAttachment`：

| 字段           | 说明        |
| -------------- | ----------- |
| `id`           | 附件 ID     |
| `messageId`    | 对话消息 ID |
| `mediaAssetId` | 媒体 ID     |
| `type`         | `image`     |
| `createdAt`    | 创建时间    |

结论：

- 若只求最快落地，可一期先把附件放 `metadata`
- 若按最佳实践，建议直接建 `MediaAsset + OcrTask + ChatAttachment`

## 11. 接口设计建议

## 11.1 上传接口

不建议继续让账单图片走当前公开头像接口。

推荐新增：

### `POST /upload/chat-image`

用途：

- 上传首页对话记账用图片

返回：

- `assetId`
- `previewUrl` 或临时签名 URL
- 图片基础信息

关键点：

- 存储到私有路径，如 `chat-bills/{userId}/{yyyy}/{mm}/{uuid}.jpg`
- 不返回永久公开地址
- 使用临时签名 URL 预览

## 11.2 对话发送接口

当前：

```json
{ "content": "午饭25" }
```

推荐升级为：

```json
{
  "content": "这是昨天的晚饭",
  "attachments": [
    {
      "assetId": "asset_xxx",
      "type": "image"
    }
  ],
  "inputMode": "image_text"
}
```

### 兼容原则

- 纯文本继续兼容旧格式
- 图片或图片混输走新字段

## 11.3 OCR 处理方式

最佳实践推荐异步化，但保留同步兜底。

### 推荐方案

- `POST /chat/send` 创建用户消息
- 如果存在图片附件，则创建 OCR 任务
- 通过 SSE 或轮询刷新 assistant 消息状态

状态示例：

- `assistant.metadata.type = "ocr_processing"`
- `assistant.metadata.type = "confirm_card"`
- `assistant.metadata.type = "ocr_failed"`

### 为什么推荐异步

- 图片上传 + OCR + LLM 解析很容易超过普通文本消息耗时
- 未来如果接多家 OCR、做重试、做流式进度展示，异步更稳

### 一期可接受的折中

如果为了加快交付，一期可以先做同步处理，但要加：

- 请求超时控制
- 前端可取消
- 明确 loading 状态
- 服务端熔断和降级

## 12. 服务端编排建议

## 12.1 新增 OCR 编排层

建议在 `server/src/modules` 中新增独立 `ocr` 模块，而不是把 OCR 逻辑全部塞进 `chat.service.ts`。

推荐模块职责：

- `ocr.module.ts`
- `ocr.service.ts`
- `ocr.provider.ts`
- `ocr.types.ts`
- `ocr-normalizer.ts`
- `receipt-parser.ts`
- `payment-screenshot-parser.ts`

## 12.2 与 ChatService 的职责边界

### ChatService 负责

- 创建消息
- 调度 OCR 流程
- 调用现有账单解析/分类匹配
- 生成 assistant 消息和确认卡片

### OcrService 负责

- 调 OCR Provider
- 返回 OCR 标准化结果

### ReceiptParser 负责

- 从 OCR 文本中抽取账单关键字段

### Chat Bill Graph 负责

- 把 OCR 输出转成项目现有 `ParseResult[]`
- 继续走已有确认链路

这样可以最大程度复用现有：

- `confirmBill`
- `confirmAllBills`
- `rejectBill`
- `Bill` 入库逻辑

## 13. OCR Provider 选型建议

## 13.1 一期推荐

优先推荐接入成熟中文 OCR 云服务，候选顺序：

1. 百度 OCR
2. 阿里云 OCR
3. 腾讯云 OCR

原因：

- 对中文票据、支付截图识别成熟
- 接口稳定
- 省去本地模型部署和维护成本

## 13.2 不建议一期主用 Tesseract

Tesseract 适合作为：

- 本地实验
- 离线兜底
- 成本极低场景

但不适合作为首页核心体验一期主方案，原因是：

- 中文复杂票据准确率一般
- 拍照场景稳定性不如云 OCR
- 后处理成本高

## 13.3 多模态 LLM 的位置

推荐把多模态 LLM 放在二期或兜底链路：

- OCR 文本质量很差
- 图像是特殊账单模板
- 规则和 OCR 都无法稳定提取时

最佳实践不是“全靠多模态”，而是“结构化 OCR 主路 + 多模态兜底”。

## 14. Mobile 端最佳实践

## 14.1 图片获取

建议使用：

- 相册：`expo-image-picker`
- 拍照：`expo-image-picker` 的 camera 或 `expo-camera`

推荐一期先用统一方案，减少原生复杂度。

## 14.2 上传前预处理

客户端必须做以下处理：

- 自动纠正图片方向
- 压缩长边，避免超大原图
- 控制大小，建议 2MB 以内优先
- 去掉不必要 EXIF 信息

原因：

- 降低上传耗时
- 降低 OCR 成本
- 减少服务端处理压力

## 14.3 图片消息体验

建议增加以下 UI 状态：

- 缩略图预览
- 上传进度
- 识别中占位卡
- 失败重试按钮

## 14.4 混合输入支持

最佳实践是支持：

- 只有图片
- 图片 + 文字

示例：

- “帮我记一下这张图”
- “这是昨天打车截图”
- “这张是退款到账”

这样可以显著提升低质量图片场景的识别准确率。

## 15. 隐私与安全要求

OCR 账单图片是敏感数据，这部分必须比头像上传更严格。

## 15.1 存储策略

账单图片必须：

- 私有存储
- 不走公开桶策略
- 仅通过签名 URL 临时访问

## 15.2 日志策略

日志中禁止记录：

- 图片原始 URL
- 全量 OCR 原文
- 用户敏感支付信息

建议仅记录：

- 任务 ID
- provider
- 场景类型
- 耗时
- 成功/失败
- 置信度

## 15.3 生命周期策略

建议提供可配置保留期：

- 原图保留 7 到 30 天
- OCR 结构化结果可长期保留
- 用户确认入账后允许异步清理原图

如果产品主打隐私，可进一步支持：

- 用户确认后自动删除原图，只保留结构化结果和摘要

---

## 16. 与现有账单链路的衔接方案

## 16.1 推荐复用现有 ParseResult

OCR 最终输出应尽量对齐现有 `ParseResult[]`：

| 字段           | OCR 输出要求               |
| -------------- | -------------------------- |
| `type`         | 必须识别收入/支出          |
| `amount`       | 必须产出主金额             |
| `categoryName` | 尽量命中现有分类           |
| `categoryId`   | 通过现有分类匹配补齐       |
| `note`         | 商户名或事件摘要           |
| `date`         | 账单日期                   |
| `accountName`  | 微信/支付宝/银行卡/现金等  |
| `confidence`   | 高中低                     |
| `warning`      | 金额冲突、模糊、疑似非账单 |

## 16.2 OCR 不直接绕过确认链路

最佳实践是不建议 OCR 直接入库，除非达到非常高的稳定度。

建议一期统一策略：

- OCR 场景默认都展示确认卡片
- 即使高置信度也建议默认展示“轻确认”

原因：

- 图片识别比文本口述更容易误判金额
- 用户对图片账单更愿意“看一眼再确认”

后续再根据数据决定是否放开高置信度自动入库。

---

## 17. 监控与评估指标

OCR 记账要做好，需要从第一天就埋点。

建议重点监控：

- 图片上传成功率
- OCR 成功率
- OCR 平均耗时
- 识别后进入确认卡片比例
- 用户确认率
- 用户修改率
- 金额修改率
- 分类修改率
- 账户修改率
- 最终放弃率

关键判断标准：

- 如果金额修改率高，说明金额提取规则有问题
- 如果分类修改率高，说明分类推荐要加强
- 如果放弃率高，说明确认卡片信息不可信或交互太重

## 18. 分阶段落地建议

## 18.1 Phase 1：可用版

目标：

- 首页支持相册上传和拍照
- 支持支付成功截图、订单详情截图、小票图
- OCR 后进入确认卡片
- 用户确认后入库

范围：

- 新增私有账单图片上传接口
- `chat/send` 支持附件
- 新增 OCR 模块
- 结构化结果映射到现有 `ParseResult`
- 首页展示图片消息和 OCR loading 状态

## 18.2 Phase 2：增强版

目标：

- 增加异步 OCR 任务和进度流
- 增加多场景解析模板
- 引入历史商户分类映射
- 增加图片 + 文字混合理解
- 增加多模态 LLM 兜底

## 18.3 Phase 3：优化版

目标：

- 自动学习常见商户分类
- OCR 质量评分与重试
- 原图清理策略
- 收据商品明细展开
- 账单证据回看

## 19. 最终推荐方案

结合当前项目状态，最优落地方案如下：

### 方案结论

1. 继续沿用“首页对话消息 -> 确认卡片 -> 入库”的现有主链路。
2. 新增独立 OCR 能力层，不直接把 OCR 逻辑塞进现有文本解析逻辑。
3. 账单图片必须走私有存储，不能复用当前公开头像上传策略。
4. 一期主路径采用“专业 OCR + 规则提取 + LLM 补全 + 置信度评估”混合方案。
5. OCR 结果一期统一进入确认卡片，不建议直接自动入库。
6. 默认一张票生成一笔账单，商品明细只做证据，不做多笔拆账。
7. 接口层建议把 `chat/send` 扩展为支持 `attachments`，为后续语音、图片、视频统一输入打基础。

## 22. 一句话结论

最适合当前项目的 OCR 账单识别方案，不是“把图片直接交给大模型”，而是“在现有对话记账链路上，新增私有图片上传、OCR 编排层、规则提取和置信度确认机制”，让图片识别最终仍然回到你已经做好的确认卡片体系中，这样准确率、可控性和后续扩展性都是最优的。
