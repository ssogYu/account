#### 在mobile里面expo 使用什么本地化存储方案

1. 目前expo 提供的本地化存储方案有

- AsyncStorage
- MMKV

  ```
  适用：简单键值对、高频读写（状态、设置、缓存）
  目前，MMKV 是 React Native 中性能最强大的键值存储方案。它是微信团队开源的底层组件，使用 C++ 编写，速度比传统的 AsyncStorage 快 30 倍以上，且支持完全同步调用（无需 await）
  ```

- expo-secure-store

  ```
  适用：敏感信息（Token、密码、密钥）、JWT Tokens、用户密码、支付相关的敏感信息
  
  无论你使用什么作为主存储，只要涉及安全数据，都必须使用它。
  优势： 官方原生支持。在 iOS 上使用 Keychain，在 Android 上使用 EncryptedSharedPreferences/Keystore 进行硬件级加密。、
  注意事项： 只适合存储小体积字符串（通常限制在 2MB 以内），读写速度较慢，不要用来存普通业务数据
  
  ```

- expo-sqlite

  ```
  适用：结构化数据、大量离线数据（复杂查询）
  如果你在做一个离线优先（Offline-first）的应用，或者需要存储大量的列表、聊天记录，并进行复杂的查询和过滤，SQLite 是唯一解。
  优势： 支持标准 SQL 语句，能够处理极其复杂的数据关联，性能极其稳定。Expo 最近对 expo-sqlite 进行了大幅重构升级，现在支持同步 API，性能有了质的飞跃。
  
  适用场景： 聊天应用、离线笔记、大型电商目录缓存、依赖大量关系型数据的应用。
  ```

  

2. 最优方案

​	在实际的生产级 Expo 商业项目中，通常会采用**组合拳**的架构

| **存储类型**        | **推荐方案**        | **主要用途**                                      |
| ------------------- | ------------------- | ------------------------------------------------- |
| **高性能/普通状态** | `react-native-mmkv` | UI 状态、App 设置项、非敏感的用户偏好、主题配置。 |
| **极密数据**        | `expo-secure-store` | 用户的登录 Token、刷新 Token、密码。              |
| **庞大/结构化数据** | `expo-sqlite`       | 缓存的商品列表、本地消息记录、复杂的离线缓存库。  |
