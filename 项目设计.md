# AI智能记账助手 - 项目设计文档

## 一、产品概述

### 1.1 产品定位

一款以AI为核心驱动力的智能记账应用，支持移动端和桌面端，帮助个人和家庭轻松记录、分析和管理日常收支。

### 1.2 目标用户

- 有日常记账需求的个人用户
- 需要共同管理家庭财务的家庭用户
- 希望通过AI降低记账门槛的懒人用户

### 1.3 核心价值

- **零门槛记账**：AI对话式记账，说话即记录
- **智能分类**：自动识别消费类型，无需手动选择
- **家庭协作**：多人共享账本，家庭财务透明化
- **数据洞察**：可视化图表让消费习惯一目了然

### 1.4 设计原则

- **统一入口**：首页即AI对话界面，所有记账都通过对话完成，没有独立的弹窗或表单
- **渐进智能**：AI高置信度直接入库，低置信度才需确认，减少交互步骤
- **简单优先**：第一期不做账户余额管理，账户仅作为标签标记资金来源
- **数据归一**：加入家庭组后所有账单自动归属家庭组，通过成员 tag 区分，不做可见性控制

## 二、第一期功能设计

### 2.1 记账模块（AI对话式记账）

> 核心思路：首页就是AI对话界面，没有独立的"记一笔"按钮或弹窗。用户和AI助手对话，AI理解意图后生成账单确认卡片，用户确认即入库。一切记账都通过对话完成。

**对话式记账流程：**

```
首页 = AI对话界面
    ↓
用户在对话框输入内容
    ├── "午饭25" → AI解析 → 对话中展示确认卡片 → 用户确认 → 入库
    ├── "昨天打车花了30" → AI解析 → 对话中展示确认卡片 → 用户确认 → 入库
    ├── "星巴克" → AI补全金额+分类 → 对话中展示确认卡片 → 用户确认 → 入库
    ├── 语音输入 → 语音转文字 → 同上流程
    ├── 上传图片 → OCR识别 → 对话中展示识别结果 → 用户确认 → 入库
    └── "这个月吃饭花了多少" → AI查询统计 → 对话中展示结果
```

**账单字段：**
| 字段 | 说明 | 必填 |
|------|------|------|
| 金额 | 收支金额 | 是 |
| 类型 | 收入/支出 | 是 |
| 分类 | 如餐饮、交通、工资等 | 是（提供默认分类） |

**分类体系设计：**

- 系统提供默认分类（餐饮、交通、购物、娱乐、居住、医疗、教育、工资、理财等）
- 用户可自定义分类，但默认分类不可删除（保证统计一致性）
- 家庭组共享统一分类体系，确保汇总统计时分类对齐

**设计要点：**

- 首页就是对话界面，没有"记一笔"按钮，没有弹窗，没有表单
- 所有记账都通过对话完成，AI理解用户意图，生成确认卡片
- 确认卡片是核心交互：AI解析后展示结构化结果，用户一键确认或修改
- 高置信度时确认卡片默认选中，用户可直接点确认；低置信度时需用户补充信息
- 对话框支持文字输入、语音输入（长按🎤录音）、图片上传（📎）
- 语音按钮常驻输入框右侧，长按录音，松开识别，支持实时语音转文字
- 今日收支卡片浮动在对话区顶部，记账后实时更新，提供即时反馈
- 本地缓存，断网也能记账

### 2.2 AI智能记账

> 核心思路：AI不是独立功能，而是记账流程中的智能增强层。LLM负责语义理解，分类推荐优先本地匹配、兜底LLM，兼顾体验和成本。

**AI处理策略（渐进智能）：**
| 置信度 | 条件 | 处理方式 |
|--------|------|----------|
| 高 | 金额明确 + 分类明确（如"午饭25"） | 直接入库，账单标记"AI自动" |
| 中 | 金额明确但分类模糊 | 入库但弹出分类选择，用户一键确认 |
| 低 | 信息不完整或歧义 | 展示解析结果，用户编辑后确认 |

**AI能力分层架构：**

记账过程中LLM的能力不是"一次调用全包"，而是按职责分层，能本地解决的不调LLM：

```
用户输入 "星巴克38"（文字/语音转文字/OCR提取文字）
         ↓
┌─────────────────────────────────────┐
│  Step 0: 输入预处理                  │
│  语音 → ASR转文字                    │
│  图片 → OCR提取文字                  │
│  文字 → 直接传入                     │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  Step 1: LLM 语义解析（必须LLM）     │
│  提取结构化信息：                     │
│  金额=38, 备注=星巴克, 日期=今天     │
│  分类=待定                           │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  Step 2: 分类推荐（三层策略）        │
│                                     │
│  Layer 1: 本地关键词匹配（0ms）      │
│  ┌─────────────────────────────┐    │
│  │ 关键词 → 分类映射缓存表      │    │
│  │ "星巴克" → 餐饮 ✓ 命中！    │    │
│  │ "滴滴" → 交通               │    │
│  │ "地铁" → 交通               │    │
│  │ 命中 → 直接返回，不调LLM    │    │
│  └─────────────────────────────┘    │
│               │ 未命中               │
│               ↓                     │
│  Layer 2: 向量相似匹配（<50ms）    │
│  ┌─────────────────────────────┐    │
│  │ PGVector查询相似账单embedding│    │
│  │ "星巴克"≈"瑞幸"→ 餐饮 ✓    │    │
│  │ 语义相似度>0.8 → 命中！     │    │
│  │ 命中 → 直接返回，不调LLM    │    │
│  └─────────────────────────────┘    │
│               │ 未命中               │
│               ↓                     │
│  Layer 3: LLM 分类推理（~1s）       │
│  ┌─────────────────────────────┐    │
│  │ "瑞幸咖啡"是什么分类？      │    │
│  │ LLM → 餐饮                  │    │
│  │ 结果回填到Layer 1缓存表     │    │
│  └─────────────────────────────┘    │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  Step 3: 智能补全（LLM增强）         │
│  根据历史推断缺失字段：              │
│  - 用户常记"星巴克"→ 自动补全分类   │
│  - 用户习惯午餐记25-35 → 异常提醒   │
│  - "房租"→ 推测是居住+周期性支出    │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│  Step 4: 置信度评估 + 异常检测       │
│  评估整体解析可靠性：                │
│  高 → 直接入库                       │
│  中 → 需确认分类                     │
│  低 → 需用户补充                     │
│  异常 → 弹出提醒                     │
└─────────────────────────────────────┘
```

**各步骤LLM使用情况：**

| 步骤              | 是否必须LLM | 说明                                        |
| ----------------- | ----------- | ------------------------------------------- |
| Step 0 输入预处理 | 否          | 语音用ASR，图片用OCR，文字直接传入          |
| Step 1 语义解析   | 是          | 自然语言→结构化数据，只有LLM能做            |
| Step 2 分类推荐   | 否（兜底）  | 80%+请求在Layer 1/2本地解决，仅新场景调LLM  |
| Step 3 智能补全   | 部分        | 周期性识别、异常检测可用规则；语义补全需LLM |
| Step 4 置信度评估 | 是          | 需LLM判断解析结果的可靠性                   |

**LLM能力详细说明：**

0. **输入预处理**（非LLM）
   - 语音输入：ASR语音转文字，支持实时流式识别，用户说话过程中即可展示转写文字
   - 图片输入：OCR提取文字，再传入LLM解析
   - 文字输入：直接传入，无需预处理

1. **语义解析**（必须LLM）
   - 从自然语言中提取：金额、备注、日期、收入/支出类型
   - 批量解析：一次输入多笔消费，如"午饭25，打车18，买水5"，AI拆分为多条账单
   - OCR图片识别后，LLM从OCR文本中提取结构化信息

2. **分类推荐**（本地优先，LLM兜底）
   - Layer 1 关键词表：系统预置200+常见消费关键词映射，用户高频消费自动补充
   - Layer 2 向量相似匹配：PGVector查询用户历史账单embedding，语义相似度>0.8即命中（如"瑞幸"≈"星巴克"→餐饮），比精确匹配更智能
   - Layer 3 LLM推理：仅全新消费场景触发，结果自动回填缓存+生成embedding写入PGVector
   - 分类推荐结果带置信度，高置信度自动填入，低置信度需用户确认

3. **智能补全**（规则+LLM混合）
   - 金额补全：用户只输入"午饭"，LLM根据历史推断金额范围（如¥20-35），展示建议值
   - 周期性识别：规则检测"每月固定日期+固定金额"→ 提示设为定期账单
   - 账户推测：用户习惯用微信付餐饮→ 自动推荐账户

4. **异常检测**（规则+LLM混合）
   - 规则层：金额超出该分类历史均值3倍 → 标记异常
   - LLM层：语义判断，"请同事吃饭500"是合理聚餐，"午饭500"可能是多输了一个0
   - 异常时弹出提醒："这笔餐饮¥500，是否金额有误？"

- 所有记账交互都在对话中完成，没有独立弹窗或表单页面

- AI解析失败时在对话中提示用户补充信息，不阻断操作
- 分类推荐缓存命中率目标：首周50%+，一个月后80%+（随用户使用自动学习）
- 异常检测阈值基于用户个人历史，不做全局统一标准
- 对话支持非记账查询，如"这个月花了多少"，AI直接返回统计结果

### 2.4 家庭组模块

> 核心思路：家庭组极简设计，加入后所有账单都属于家庭组，统计页按成员区分。家庭组没有独立页面，在我的页中作为设置项，点击弹窗操作。

**核心功能：**

- 创建家庭组，生成邀请码/链接
- 成员通过邀请码加入
- 加入后，该成员所有账单自动归属家庭组

**账单归属设计：**

- 用户未加入家庭组 → 账单仅属于个人
- 用户加入家庭组 → 所有账单自动归属家庭组，无需手动标记
- 统计页通过成员 tag 区分每笔消费是谁的，支持按成员筛选
- 统计页支持查看"全家汇总"和"个人统计"两个维度

**角色与权限：**
| 角色 | 权限 |
|------|------|
| 组长 | 管理成员、解散家庭组 |
| 成员 | 记账、查看汇总、退出家庭组 |

**交互方式：**

- 家庭组没有独立页面，在我的页中作为"家庭组"设置项
- 点击"家庭组" → 弹出弹窗，展示家庭组信息和操作
- 弹窗内操作：查看成员列表、邀请成员（复制邀请码）、退出/解散家庭组

**设计要点：**

- 极简设计，不做账单可见性控制，加入家庭组后所有账单共享
- 统计页是家庭数据的主要展示入口，通过成员 tag 和筛选实现个人/家庭视图切换
- 家庭组管理是低频操作，弹窗即可满足，不需要独立页面

### 2.5 账单管理模块

### 2.6 预算与提醒模块

> 核心思路：预算和提醒本质上都是"在合适时机提醒用户"，合并为一个通知体系。

**预算功能：**

- 设置月度总预算（如每月支出不超过5000元）
- 可选设置分类预算（如餐饮不超过1500元）
- 首页/统计页展示预算进度条，已用/剩余一目了然
- 每日记账后自动更新预算进度

**提醒策略（统一通知中心）：**
| 触发条件 | 提醒内容 | 优先级 |
|----------|----------|--------|
| 每日定时（用户设定） | "今天还没记账哦，昨天你花了¥128" | 普通 |

## 三、数据模型设计

### 3.1 核心实体

```
User（用户）
├── id, phone, nickname, avatar, created_at

Account（账户标签）
├── id, user_id, name, type(微信/支付宝/现金/银行卡)
  注：仅作为账单标记，不维护余额；已移除icon字段，前端以tag样式展示

Category（分类）
├── id, user_id(nullable), name, type(收入/支出), icon, is_system
  注：is_system=true 为系统默认分类，user_id=null；用户自定义分类 user_id 非空

Bill（账单）
├── id, user_id, family_id(nullable), amount, type(收入/支出),
│   category_id, account_id, date, remark, source(手动/AI),
│   ai_confidence(AI置信度，仅AI记账时有值),
│   created_at, updated_at
  注：family_id 非空时表示该账单属于家庭组；用户加入家庭组后，所有账单自动关联 family_id

Budget（预算）
├── id, user_id, category_id(nullable), amount, month

Family（家庭组）
├── id, name, creator_id, invite_code, created_at

FamilyMember（家庭成员）
├── id, family_id, user_id, role(组长/成员), joined_at

CategoryKeyword（分类关键词缓存，AI分类推荐Layer 1）
├── id, keyword, category_id, source(系统预置/用户学习), hit_count, updated_at
  注：系统预置200+关键词，用户高频消费自动补充，LLM推理结果回填

BillEmbedding（账单向量，AI分类推荐Layer 2）
├── id, user_id, bill_id(nullable), remark, category_id,
│   embedding(vector(1536)), created_at
  注：每笔AI记账后生成embedding写入，用于语义相似匹配；向量维度取决于embedding模型

ChatMessage（对话记录，AI记账交互上下文）
├── id, user_id, role(用户/AI), content, bill_id(nullable), created_at
  注：仅AI记账时产生，手动记账不产生对话记录；7天自动清理

BillParse（AI解析结果）
├── id, chat_message_id, bill_id, confidence(置信度), parse_detail(JSON)
  注：parse_detail存储LLM原始解析结果，便于调试和优化Prompt
```

**数据模型关系说明：**

- Bill 设 family_id：用户加入家庭组后，所有账单自动关联 family_id，统计页通过 family_id + user_id 实现家庭汇总和成员筛选
- CategoryKeyword 是分类推荐三层策略的 Layer 1 缓存表，系统预置+动态学习
- BillEmbedding 是分类推荐 Layer 2 的向量表，PGVector扩展，每笔AI记账后自动生成embedding
- ChatMessage 与 Bill 是松耦合：AI记账时关联，手动记账时无对话记录
- BillParse.parse_detail 存JSON：保留LLM原始输出，方便后续分析Prompt效果和迭代优化

```typescript
// ai/llm.factory.ts - Provider工厂，统一创建模型实例
@Injectable()
export class LlmFactory {
  constructor(private readonly configService: ConfigService) {}

  createModel(task: 'parse' | 'recommend' = 'parse'): BaseChatModel {
    const provider = this.configService.get('LLM_PROVIDER', 'glm');
    const config = PROVIDER_CONFIG[provider];

    switch (provider) {
      case 'glm':
        return new ChatZhipuAI({
          model: config.models[task],
          temperature: config.temperature[task],
          zhipuAIApiKey: this.configService.get('GLM_API_KEY'),
        });
      case 'openai':
        return new ChatOpenAI({
          modelName: config.models[task],
          temperature: config.temperature[task],
          openAIApiKey: this.configService.get('OPENAI_API_KEY'),
        });
      case 'qwen':
        return new ChatTongyi({
          model: config.models[task],
          temperature: config.temperature[task],
          tongyiApiKey: this.configService.get('QWEN_API_KEY'),
        });
      case 'deepseek':
        return new ChatDeepSeek({
          model: config.models[task],
          temperature: config.temperature[task],
          apiKey: this.configService.get('DEEPSEEK_API_KEY'),
          modelKwargs: { thinking: { type: 'disabled' } },
        });
      default:
        throw new Error(`Unsupported LLM provider: ${provider}`);
    }
  }
}
```

**为什么第一期就用LangGraph.js：**

- LangGraph.js 基于图的状态机模型，天然支持条件分支（置信度高/中/低走不同路径），比线性Chain更贴合"渐进智能"设计
- 线性Chain处理条件分支需要嵌套判断，代码复杂度反而更高；LangGraph.js 用节点+边声明流程，逻辑更清晰
- 后续扩展（多轮对话、消费建议Agent）无需重构，直接在图上增加节点和边即可
- LangGraph.js 是 LangChain 官方出品，与 LangChain.js 无缝集成，学习成本仅多一个图定义概念

```typescript
import { Annotation } from "@langchain/langgraph";

// 定义图的状态结构
const BillParseState = Annotation.Root({
  input: Annotation<string>,             // 用户原始输入
  inputType: Annotation<string>,         // 数字/自然语言/图片/语音
  amount: Annotation<number | null>,     // 解析出的金额
  remark: Annotation<string>,            // 备注
  date: Annotation<string>,              // 日期
  type: Annotation<string>,              // 收入/支出
  categoryId: Annotation<string | null>, // 分类ID
  categoryConfidence: Annotation<number>,// 分类置信度 0-1
  accountId: Annotation<string | null>,  // 推荐账户
  confidence: Annotation<string>,        // 整体置信度: high/mid/low
  isAbnormal: Annotation<boolean>,       // 是否异常消费
  abnormalReason: Annotation<string>,    // 异常原因
});

// Node 0: 输入预处理（语音转文字/OCR）
async function preprocessInput(state: typeof BillParseState.State) {
  if (state.inputType === "voice") {
    // 语音转文字：优先用系统原生ASR，失败降级云端
    const text = await asrService.recognize(state.input);
    return { input: text, inputType: "natural_language" };
  }
  if (state.inputType === "image") {
    // OCR提取文字
    const text = await ocrService.extract(state.input);
    return { input: text, inputType: "natural_language" };
  }
  return {}; // 文字/数字直接传入
}

// Node 1: 语义解析
async function parseInput(state: typeof BillParseState.State) {
  const model = llmFactory.createModel("parse");
  const result = await model.invoke(/* 解析Prompt */);
  return { amount: result.amount, remark: result.remark, ... };
}

// Node 2: 分类推荐（三层策略）
async function classifyCategory(state: typeof BillParseState.State) {
  // Layer 1: 本地关键词匹配
  const keywordMatch = await keywordRepo.findByKeyword(state.remark);
  if (keywordMatch) return { categoryId: keywordMatch.categoryId, categoryConfidence: 0.95 };

  // Layer 2: PGVector向量相似匹配
  const embeddings = new OpenAIEmbeddings(); // 或其他embedding模型
  const queryEmbedding = await embeddings.embedQuery(state.remark);
  const similarBill = await prisma.$queryRaw`
    SELECT category_id, 1 - (embedding <=> ${queryEmbedding}::vector) AS similarity
    FROM bill_embedding
    WHERE user_id = ${userId} AND 1 - (embedding <=> ${queryEmbedding}::vector) > 0.8
    ORDER BY similarity DESC LIMIT 1
  `;
  if (similarBill.length > 0) {
    return { categoryId: similarBill[0].category_id, categoryConfidence: similarBill[0].similarity };
  }

  // Layer 3: LLM推理
  const model = llmFactory.createModel("parse");
  const llmResult = await model.invoke(/* 分类Prompt */);
  // 回填Layer 1缓存 + 生成embedding写入PGVector
  await keywordRepo.create({ keyword: state.remark, categoryId: llmResult.categoryId, source: "用户学习" });
  const remarkEmbedding = await embeddings.embedQuery(state.remark);
  await prisma.billEmbedding.create({ data: { userId, embedding: remarkEmbedding, categoryId: llmResult.categoryId } });
  return { categoryId: llmResult.categoryId, categoryConfidence: llmResult.confidence };
}

// Node 3: 智能补全
async function smartComplete(state: typeof BillParseState.State) {
  // 规则补全：账户推测、周期性识别
  const accountSuggestion = await suggestAccount(state, userId);
  return { accountId: accountSuggestion };
}

// Node 4: 置信度评估 + 异常检测
async function evaluateConfidence(state: typeof BillParseState.State) {
  // 规则层异常检测
  const isAbnormal = await checkAbnormal(state, userId);
  // LLM置信度评估
  const confidence = await assessConfidence(state);
  return { confidence, isAbnormal };
}

// 条件边：按置信度路由
function routeByConfidence(state: typeof BillParseState.State): string {
  if (state.isAbnormal) return "confirm_abnormal";
  if (state.confidence === "high") return "auto_save";
  if (state.confidence === "mid") return "confirm_category";
  return "need_supplement";
}

// 构建图
const graph = new StateGraph(BillParseState)
  .addNode("preprocess", preprocessInput)
  .addNode("parse", parseInput)
  .addNode("classify", classifyCategory)
  .addNode("complete", smartComplete)
  .addNode("evaluate", evaluateConfidence)
  .addEdge("__start__", "preprocess")
  .addEdge("preprocess", "parse")
  .addEdge("parse", "classify")
  .addEdge("classify", "complete")
  .addEdge("complete", "evaluate")
  .addConditionalEdges("evaluate", routeByConfidence, {
    auto_save: END,
    confirm_category: END,
    need_supplement: END,
    confirm_abnormal: END,
  });

const app = graph.compile();
```

**节点与LLM调用关系：**

| Node               | 是否调LLM  | 调用时机                             |
| ------------------ | ---------- | ------------------------------------ |
| Node 0 输入预处理  | 否         | 语音用ASR，图片用OCR，文字直接传入   |
| Node 1 语义解析    | 是         | 每次自然语言/语音/图片输入必调       |
| Node 2 分类推荐    | 否（兜底） | Layer 1/2命中不调LLM，仅Layer 3调    |
| Node 3 智能补全    | 部分       | 周期性/账户推测用规则，金额补全用LLM |
| Node 4 置信度+异常 | 是         | 置信度评估需LLM，异常检测规则优先    |

**建议：**

- LLM调用做超时和降级处理，超时3秒降级为手动模式
- 解析结果缓存：相同/相似输入命中缓存，减少LLM调用成本
- Prompt模板版本化管理，方便迭代优化
- 流式输出用 Socket.IO 或 SSE（Server-Sent Events），前端实时展示AI解析过程
- 不同任务用不同模型：解析用轻量快速模型（glm-4-flash/gpt-4o-mini），推荐用强推理模型（glm-4/gpt-4o），平衡成本和效果

**多消费解析实现（已实现）：**

当用户一次输入多笔消费（如"午饭25，打车15，奶茶8"），系统通过以下机制处理：

1. **LLM Schema 层**：`ParseOutputSchema` 的 `bills` 字段定义为 `z.array(SingleBillSchema)`，LLM 一次返回多笔消费数组
2. **LangGraph 状态层**：`BillState` 的 `bills` 字段为 `BillItem[]`，支持多笔消费在图中流转
3. **后端服务层**：`ChatService` 的 `confirmBill` 支持按 `billIndex` 确认单笔，`confirmAllBills` 支持一次确认全部
4. **前端交互层**：`ConfirmCard` 组件自动识别单笔/多笔模式，多笔时展示"识别到N笔消费"头部、单笔确认按钮和"全部确认"按钮
